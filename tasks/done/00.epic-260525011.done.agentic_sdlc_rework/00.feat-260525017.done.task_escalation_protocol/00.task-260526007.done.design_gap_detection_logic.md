# Design gap detection in implement task skill

# Design gap detection in implement task skill

## Problem statement

`/agn:implement task` today proceeds from the task body straight to coding. If the task body's scope or acceptance criteria leave a meaningful design question open, the implementer either improvises or stops without a structured handoff. Both corrupt Design/Implementation separation. The skill needs a checkpoint: after detailed design, before coding, detect whether the task body answers the design questions the implementer would otherwise have to guess on.

## Scope

In scope:
- Edit `plugins/agn/skills/implement/SKILL.md`, `$0 = task` branch.
- Insert a "Design gap detection" sub-step between current step 2 (Read and understand → detailed design first) and step 3 (Architecture compliance checkpoint).
- Detection prompts: list ambiguities or missing information surfaced during detailed design; classify by suspected upstream level (task body | feature scope | epic design | product architecture).
- If gaps found, halt — do not proceed to architecture checkpoint or coding.

Out of scope:
- The gap-log file format and storage (sibling task).
- The routing message format (sibling task).

## Acceptance criteria

- `$0 = task` branch has a "Design gap detection" sub-step before architecture checkpoint.
- The step instructs the implementer to enumerate ambiguities surfaced during detailed design.
- Each ambiguity is classified by suspected upstream level.
- If any gap is found, the skill halts and routes (handed off to sibling tasks' format/message).
- If no gaps, skill proceeds normally to step 3.

## Quality gates

- Skill body parses; YAML frontmatter intact.
- New step is explicit and easy to spot during reading.

## Summary

### Steps completed

1. Inserted a "Design gap detection" step (now step 3) between the existing "Read and understand" (step 2, includes detailed design) and "Architecture compliance checkpoint" (renumbered to step 4).
2. The new step instructs the implementer to enumerate ambiguities surfaced during detailed design and classify each by suspected upstream level (`task` body, `feature` scope, `epic` design, product `architecture`).
3. On any gap, the step routes to the **Design gap escalation protocol** section and halts. With no gaps, it falls through to architecture checkpoint.
4. Renumbered subsequent steps (Architecture compliance → 4, Execute → 5, Complete → 6, Documentation → 7).

### Changes made

Modified:
- `plugins/agn/skills/implement/SKILL.md` (`$0 = task` Execution steps section)

### Notable decisions

- **Detection happens after detailed design, not before.** A gap is "what design info would I have to guess" — the implementer needs to attempt the design to know what's missing. Putting detection earlier would surface only the obvious gaps.
- **Suspected level is mandatory per gap.** Forcing classification at detection means the routing message can name the right upstream skill. Without it, the user gets a vague "go fix something upstream."
