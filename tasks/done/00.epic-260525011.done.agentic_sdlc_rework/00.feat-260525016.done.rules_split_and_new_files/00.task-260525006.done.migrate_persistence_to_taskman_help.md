# Migrate persistence rules into taskman.sh help output


## Problem statement

`rules/task-management.md` mixes composition rules (frontmatter shapes, body sections, summary template) with persistence rules (storage layout, lifecycle preconditions, CLI surface, validation). Persistence rules duplicate what `taskman.sh help` could authoritatively own. Today every skill that loads `task-management.md` loads both halves even when it needs only one.

## Scope

In scope: expand `scripts/taskman.sh help` output to include the persistence content currently in `rules/task-management.md` — storage layout (folder ↔ status invariant; epic and feature flat folders; task lifecycle subfolders), naming (`YYYYMMDD[_NN]_<slug>.md`; slug regex), lifecycle preconditions (feature close requires all member tasks done; epic close requires all member features done), migration of legacy files (warnings vs errors). Output should remain readable on terminal — single screen per section ideally; clear headings.

Out of scope: deleting `task-management.md` (covered by `split_task_management_rules`); authoring composition content in `task-composition.md` (covered by `split_task_management_rules`).

## Acceptance criteria

- `./scripts/taskman.sh help` includes the persistence content from `rules/task-management.md`.
- A reader of the help output can answer: where do task files live? what is the slug format? when can a feature close? when can an epic close? what do legacy / stale references trigger?
- The help output remains scannable (sectioned, not a wall of text).

## Quality gates

- `./plugins/agn/scripts/taskman.sh validate` exits 0.
- `./plugins/agn/scripts/taskman.sh help | wc -l` returns a reasonable count (target: 100-200 lines; not a 500-line dump).
- Manual review of the help output for readability.

## Summary

### Steps completed

1. Read the `usage()` function in `taskman.sh` to confirm current shape.
2. Rewrote `usage()` to embed two top-level sections: **COMMANDS** (existing CLI surface, reorganized) and **PERSISTENCE MODEL** (storage layout, naming, lifecycle preconditions, validation behavior, approval, git relationship).
3. Pointed the trailing `See:` line at `rules/task-composition.md` (composition reference) instead of the soon-to-be-deleted `rules/task-management.md`.
4. Verified output: `taskman.sh help | wc -l` = 102 lines, within the 100-200 target.

### Changes made

Modified `plugins/agn/scripts/taskman.sh`:
- Replaced the body of `usage()` (formerly ~40 lines) with a 100-line authoritative reference.

### Notable decisions or deviations

- **Split the help output with double-rule section banners** (`==========` / heading / `==========`) so COMMANDS and PERSISTENCE MODEL stand apart at a glance.
- **`## Summary` requirement included under LIFECYCLE PRECONDITIONS**, not under composition. The rule is a precondition on the `done` transition (lifecycle), even though the summary's body shape is composition. Reasoning: a reader hitting `feature close` failure needs to know the summary precondition without context-switching to a separate composition doc.
- **APPROVAL and RELATIONSHIP TO GIT sections kept** rather than dropped. They are short, persistence-adjacent, and read naturally next to the lifecycle preconditions. Dropping them would force a future reader to hunt elsewhere for the same context.
- **`task-management.md` deletion deferred** to `split_task_management_rules` per the out-of-scope note. As of this task's close, `task-management.md` still exists with both halves; the persistence half is now duplicated in `taskman.sh help`. The duplication is resolved when the split task closes.
