# ui command with local web server, nested tree, ready/blocked/recent, mermaid graph

## Problem statement

CLI lists are good for agents but not for humans browsing a 40+ item tree. A local read-only web UI gives a fast visual surface for browse / search / dependency-graph review without committing to a daemon or external service.

## Scope

**In:** `taskman ui` subcommand starting a local HTTP server on `localhost:8080` (configurable via `--port`); single HTML page rendering five sections (nested tree, ready, blocked, recently done from last 7 days by file mtime, Mermaid dependency graph); client-side JavaScript search filter; side-panel preview of item body via `/api/show/<id>`; `taskman ui --stop` to kill a running server via PID lockfile; Ctrl+C in the foreground also stops it; stdlib `http.server.ThreadingHTTPServer` (no new deps).

**Out:** Editing through the UI (read-only); authentication; multi-user; websocket live updates (refresh works); per-item edit links.

## Acceptance criteria

1. `taskman ui` starts a local server and opens the default browser to the rendered page.
2. The page renders five sections: nested tree, ready, blocked, recently completed (last 7 days by file mtime), Mermaid dependency graph.
3. A search box filters items across all sections client-side with no page reload.
4. Clicking an item opens its body in a side panel via `/api/show/<id>` (returns JSON).
5. Refreshing the page reflects current disk state.
6. `taskman ui --stop` reads the PID lockfile and terminates the running server; Ctrl+C also stops cleanly.
7. The Mermaid dependency graph renders in-browser (Mermaid loaded from CDN, no server-side image generation).
8. Library choice documented + justified: stdlib `http.server.ThreadingHTTPServer` + inline HTML, no new runtime dependencies.
9. pytest covers data-aggregation functions; a smoke test starts the server on an ephemeral port, fetches `/`, asserts HTTP 200 and presence of key section markers.

## Quality gates

- pytest passes.
- Read-only command (no writes to the tree).
- No new third-party dependencies (stdlib `http.server` + `urllib` only).
- Smoke test exercises the server end-to-end.

### Notable decisions
- **Library choice: stdlib `http.server.ThreadingHTTPServer` + inline HTML/JS.** Zero new runtime deps. The UI is read-only, single-user, single-page, and serves a small data volume — FastAPI's features (async, validation, OpenAPI) buy nothing here while adding 3 deps. Mermaid loaded via CDN client-side, no server-side image work.
- **"Recently done" uses file mtime** as proxy for completion date (no `closed_at` field tracked in the new schema). Cutoff: 7 days. Pragmatic and accurate for typical workflows.
- **PID lockfile** at `tempfile.gettempdir() / 'taskman_ui.pid'`. `--stop` reads it and signals SIGTERM. Cleanly removed on Ctrl+C / serve_forever exit.
- **HTML escape via `html.escape`** for all user-controlled text (titles, slugs, IDs, mermaid labels). Verified by `test_render_page_escapes_html` injecting `<script>`.
- **Smoke test uses ephemeral port (port=0)** + `threading.Thread` + `urllib.request` — no external `requests` dep, no port conflicts.
