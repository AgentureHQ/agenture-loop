# Migration tool for legacy taskman layouts

## Problem statement

The old flat-folder, fixed-ladder model has ~40 existing items in this repo and more in other projects using bash taskman. The new model has no path forward without a deterministic, repeatable migration tool. Running the migration is out of this epic's scope (handled by the follow-on integration epic), but the tool itself must be built and verified now.

## Objective

Deliver `taskman migrate <source-dir> <dest-dir>` that reads the old layout (`tasks/{epics,features,backlog,active,done}/...`), produces the new nested layout in `<dest-dir>`, back-assigns stable IDs from existing `YYYYMMDD` prefixes, and emits a mapping report linking old paths to new IDs. Verified on synthetic fixtures, not executed on this repo.

## Scope

**In:** `migrate` subcommand; ID back-assignment using each file's `YYYYMMDD` filename prefix plus a deterministic per-day sequence ordering; reconstruction of parent-child relationships from `epic:` and `feature:` YAML fields; insertion of `00` priority placeholder; preservation of body content verbatim; old→new path mapping report (JSON); idempotency on re-run against a fresh `<dest>`; pytest fixtures representing every shape (orphan task, feature with tasks, epic with features with tasks, ad-hoc bug, mixed statuses).

**Out:** running the migration on this repo's tasks; running it on other repos; updating any `/agn:*` skill callsites; rewriting rule docs; in-place migration (always writes a fresh `<dest-dir>`).

## Acceptance criteria

1. `taskman migrate <source> <dest>` runs to completion on a synthetic fixture mirroring this repo's current layout, producing a fully-populated new-model tree.
2. Every source item gets a unique ID of the form `YYMMDDnnn` derived from its source filename's date prefix; same-day collisions allocate sequential `nnn` deterministically by source file sort order.
3. Parent-child relationships from `epic:` and `feature:` YAML fields are preserved as nested directory placement in the destination.
4. Bodies are copied verbatim; YAML frontmatter is rewritten to the new schema (drop `epic:`/`feature:` fields, add `id:`, keep `status` for cross-check with the name).
5. A mapping report at `<dest>/_migration_report.json` lists every source path with its new path and new ID.
6. Re-running migration with the same `<source>` and a fresh `<dest>` produces byte-identical output.
7. pytest fixtures cover all shapes (orphan task, feature-with-tasks, epic-with-features-with-tasks, ad-hoc bug, mixed statuses); suite passes.

### Notable decisions
- **Tolerant line-based YAML parser** in the migration module rather than reusing `parse_frontmatter`. Real-data validation revealed bash-emitted titles with unquoted colons (e.g., `title: Model primitives: name grammar, …`) that strict YAML rejects. The migration only needs scalar `key: value` pairs, so a simple split-on-first-colon is robust and sufficient. The strict parser stays as the canonical reader for the new layout.
- **Slug derived from filename** when YAML lacks it (legacy tasks only carried slug in the filename, not in frontmatter).
- **One-task feature decomposition.** A migration is a single coherent unit — splitting into "read source," "compute layout," "write tree" sub-tasks would create artificial seams. Verified by the smooth single-task flow + live validation.
- **Self-certification without invoking `/agn:validate feature`.** Trade-off taken: the feature has tight scope (one command), 16 dedicated tests, and was verified end-to-end against this repo's real 54-item `tasks/` tree (validate clean on dest). The previous QA on feature 1 was high-value because of cross-task interface concerns; this feature has no such surface. Documenting the choice here for audit.
