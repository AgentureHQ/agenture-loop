# Taskman Python rewrite around uniform recursive work-item model

## Problem statement

The bash `taskman.sh` and its underlying model have hit three compounding limits:

1. **Flat folders are unmanageable.** Forty-plus epics, features, and tasks sit in flat directories. Browsing, finding, and reasoning about hierarchy require grepping or opening files. The folder-as-status convention for tasks does not extend to epics and features, which sit in fixed folders with status only in YAML â€” two rules in one model.
2. **The fixed `product â†’ epic â†’ feature â†’ task` ladder does not match real work shapes.** Sometimes a one-off task has no parent; sometimes an epic decomposes straight into tasks without features; sometimes a "feature" turns out small enough to collapse into a task. The fixed ladder forces awkward placeholders.
3. **Bash is at its ceiling.** YAML parsing via awk handles only flat scalars (lists like `depends_on: [a, b]` break it), recursive graph queries would triple the code, and 1500+ lines of bash has no realistic test substrate. Cross-project demand (numio, infra, static-sites) cannot be served by a script that is hard to install and harder to extend.

A port at parity is low value â€” the model itself is what needs to change.

## Objective

Rewrite taskman in Python around a **uniform recursive work-item model**. Replace the fixed four-tier ladder with one work-item type that can nest arbitrarily. Replace flat status folders with a nested directory structure where each work item's identity (priority, ID, status, slug) is encoded in its filename or directory name. Ship as a `uv tool install`-able CLI usable by humans and agents across multiple projects. Build the migration tool to bring legacy items into the new model â€” running it on this repo is a separate follow-on epic paired with `/agn:*` skill rewiring.

## Scope

**In scope:**

- Top-level `taskman/` Python package; `pyproject.toml` for `uv tool install` and `pipx`.
- Uniform work-item model: nested directories, name grammar `[priority].<type>-<id>.[status].[slug]` (where `<type>` is `epic|feat|task`; `<id>` is the bare 9-digit `YYMMDDnnn` â€” the stable identity), ID generation (shared daily counter across types, three-digit sequence, gaps allowed), fileâ†’directory auto-conversion on child add, type-label conversion command, frontmatterâ†”name reconciliation in `validate`.
- Type labels (`epic`, `feat`, `task`) are visual only; the ladder is free (any nesting allowed). Current constraint: `task = leaf` (lifted in a future epic).
- Core CLI built on `typer`: `new`, `move`, `convert`, `list`, `show`, `close`, `validate`, `help`, `search`, `ready`, `dependents`, `waiting-on`, `worktree-status`, `ui`, `migrate`.
- YAML I/O via `ruamel.yaml` (preserves comments and formatting on round-trip).
- `--json` output on every read/list command for agent consumption.
- New schema field: `depends_on:` (list of stable item-IDs), with graph integrity validation.
- Migration command: read old layout, produce new layout, back-assign IDs from `YYYYMMDD` prefixes. Tested on synthetic fixtures.
- Local web UI for browse, search, prioritize, and dependency graph.
- pytest suite covering all commands.
- `TASKMAN_TASKS_DIR` envvar honored for cross-project use.

**Out of scope (deferred to follow-on epic `taskman_skill_integration`):**

- Running the migration on this repo's ~40 items.
- Rewriting every `/agn:*` skill callsite to the new CLI.
- Rewriting `rules/task-composition.md` and `CLAUDE.md` to describe the new model.
- Retiring or replacing `plugins/agn/scripts/taskman.sh`.

**Out of scope (deferred indefinitely):**

- Publishing to PyPI or Homebrew.
- TUI mode (web UI is the human surface).
- Linear-style features (assignees, due dates, sprints, comments, capacity).
- Multi-user, cloud, or API-server modes.
- Lifting the `task = leaf` constraint.
- A priority schema beyond the `00` filename placeholder.

## Acceptance criteria

1. `uv tool install -e ./taskman` installs the `taskman` CLI to PATH; `taskman` operates against any tasks tree selected by `TASKMAN_TASKS_DIR` or cwd.
2. The uniform work-item model is the only model implemented end-to-end: nested directories with the `[priority].<type>-<id>.[status].[slug]` name grammar (where `<type>` is `epic|feat|task` and `<id>` is the bare 9-digit `YYMMDDnnn` â€” IDs are stable across type relabel); the `task = leaf` invariant; free nesting of `epic`/`feat` labels.
3. The CLI uses `typer`; every read/list command supports `--json`; `taskman --help` and `taskman <cmd> --help` render rich help.
4. `taskman help` documents the new persistence model end-to-end and is the single source of truth in-tool.
5. All seven linked features close with passing tests.
6. The old `plugins/agn/scripts/taskman.sh` and all current `/agn:*` skill callsites are untouched in this repo; the new tool lives alongside without disrupting in-repo workflow. ### Notable cross-cutting decisions
- **Bare ID is the stable identity; type label appears in the rendered name** for visual scannability. Spec wording amended mid-flight (during feature 1 QA) to match; AC2 now reads `[priority].<type>-<id>.[status].[slug]`.
- **`task = leaf` invariant** enforced by `new`, `convert`, and `validate`. Lifting it is a future enhancement.
- **Subtree moves with root** on `move` and `close` (top-level); nested items keep their own in-name status tokens.
- **Frontmatter is the source of truth for content** (title, body); the on-disk name is a generated cache; `validate` reconciles them and errors on drift.
- **One small cross-feature fix** to `validate` during feature 2 â€” skip top-level underscore-prefixed meta files so `_migration_report.json` doesn't false-trigger `wrong_location`.
