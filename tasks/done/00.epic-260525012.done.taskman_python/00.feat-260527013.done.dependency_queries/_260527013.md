# Ready, dependents, waiting-on graph queries

## Problem statement

Once `depends_on:` exists, project managers and agents need queries to answer "what can I start now?" and "what is waiting on this?" The old `blocks`/`blocked-by` naming was semantically ambiguous and easy to invert; the new naming retires that ambiguity.

## Objective

Add three commands that walk the dependency graph with names chosen to be unambiguous:

- `taskman ready` — items in non-done status whose every `depends_on` ID points to a done item.
- `taskman dependents <id>` — non-done items that list `<id>` in their `depends_on`.
- `taskman waiting-on <id>` — non-done items that `<id>` lists in its `depends_on`.

## Scope

**In:** three commands; graph traversal logic; `--json` output mode on each; graceful handling of unknown IDs (warning to stderr, non-zero exit).

**Out:** visual dependency graph (covered in `web_ui`); bulk dependency editing; cycle detection in queries (already handled by `validate`).

## Acceptance criteria

1. `taskman ready` returns work items in non-done status whose every `depends_on` ID points to a done item. Items with empty or absent `depends_on` are included. Sorted: priority ascending, then ID.
2. `taskman dependents <id>` returns non-done items listing `<id>` in their `depends_on`.
3. `taskman waiting-on <id>` returns non-done items that `<id>` lists in its `depends_on`.
4. Every command accepts `--json`.
5. Unknown or invalid IDs produce a stderr warning and a non-zero exit code.
6. pytest covers empty graph, single-edge graph, multi-edge graph, transitive closure, missing-ID reference, and the (already-validated) cyclic-graph case to confirm queries do not infinite-loop.

### Notable decisions
- Single shared tree-walk (`_load_graph`) feeds all three commands; cheap and readable.
- Unknown deps treated as blocking for `ready` (waiting on something that doesn't exist); `validate` separately warns.
- `dependents`/`waiting-on` reject missing target with non-zero exit; `ready` is non-targeted.
